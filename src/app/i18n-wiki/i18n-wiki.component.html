Internationalization (i18n) in Angular

This section covers how to prepare your Angular application for multiple languages and locales.

What is Internationalization (i18n)?

Internationalization (often abbreviated as i18n, because there are 18 letters between 'i' and 'n') is the process of designing and preparing your application so it can be adapted to various languages and regions (locales) without engineering changes.

This involves more than just translating text. It also includes considerations for:

  • Date and time formats
  • Number formats (e.g., decimal separators, currency)
  • Pluralization rules for different languages
  • Text direction (e.g., left-to-right, right-to-left)
  • Cultural sensitivities and conventions

Localization (L10n) is the subsequent process of adapting the internationalized app for a specific locale, which includes translating text and adding locale-specific components.

Angular's Built-in i18n Approach

Angular provides a built-in internationalization framework that operates at compile time. Key characteristics include:

  • Template-based translation: You mark text in your component templates for translation.
  • One build per locale: You typically build and deploy a separate version of your application for each language/locale you support.
  • Standard translation file formats: Angular uses industry standards like XLIFF (XML Localisation Interchange File Format) or ARB (Application Resource Bundle) for translation files.
  • No runtime language switching by default: Switching languages usually means navigating to a different version of the application (e.g., hosted at example.com/en/ or example.com/fr/).

1. Marking Text for Translation in Templates

The primary way to mark static text in your templates for translation is by using the i18n attribute.


<!-- Basic usage -->
<h1 i18n>Hello World</h1>

<!-- With a description and meaning (for translators) -->
<p i18n="User welcome message on the home page@@home.welcome">
  Welcome to our application!
</p>
<!-- home.welcome is a custom ID. If not provided, Angular generates one. -->
<!-- Text after "@@" is the custom ID. Text before is description/meaning. -->

<!-- Translating attributes -->
<img src="logo.png" i18n-alt alt="Application Logo" title="Our Logo" i18n-title>

<!-- For placeholders in input elements -->
<input type="text" placeholder="Enter your name" i18n-placeholder>

When you use the i18n attribute, Angular's CLI can extract these marked texts into a translation source file.

2. Pluralization

Different languages have different rules for pluralization. Angular uses the ICU (International Components for Unicode) message format to handle this.


<!-- 'itemCount' is a component property -->
<span i18n>
  {{'{'}}itemCount, plural,
    =0 {{{'{'}}No items}}}
    =1 {{{'{'}}One item}}}
    other {{{'{'}}{{{'{'}}itemCount}} items}}
  }}
</span>

<!-- Example with offset -->
<!-- If itemCount is 0, output: "No messages." -->
<!-- If itemCount is 1, output: "You have a new message." -->
<!-- If itemCount is 5, output: "You have 4 other messages." (because of offset:1) -->
<span i18n>
  {{'{'}}itemCount, plural, offset:1
    =0 {{{'{'}}No messages.}}}
    =1 {{{'{'}}You have a new message.}}}
    other {{{'{'}}You have {{{'{'}}itemCount - 1}} other messages.}}}
  }}
</span>

3. Select (Gender or Other Conditional Messages)

The ICU message format also supports conditional messages based on a string value, often used for gender.


<!-- 'gender' is a component property (e.g., 'male', 'female', 'other') -->
<span i18n>
  {{'{'}}gender, select,
    male {{{'{'}}Invite him}}}
    female {{{'{'}}Invite her}}}
    other {{{'{'}}Invite them}}}
  }}
</span>

These are the basics of marking up your templates. Next, we'll look at translating text in component code and the build process.

4. Translating Text in Component Code ($localize)

While most text should ideally be in your templates, sometimes you need to translate strings directly in your component's TypeScript code (e.g., for dynamic messages like error notifications from a service).

Angular provides the $localize tagged message string function for this. You need to import it globally, typically in your polyfills.ts or a central script file:


// In polyfills.ts (or a similar global script file)
import '@angular/localize/init';

Once imported, you can use $localize in your component code:


// In your component.ts
// No explicit import needed in the component file itself after global init.

export class MyComponent {{'{'}}
  getWelcomeMessage(name: string): string {{'{'}}
    // Basic usage
    // return $localize \`Hello ${{'{'}}name}! Welcome to our application.\`;

    // With meaning and description (for translators)
    return $localize \`:UserGreeting|A welcome message for the user on their dashboard@@dashboard.userWelcome:Hello ${{'{'}}name}! Welcome back.\`
  }

  getErrorMessage(): string {{'{'}}
    // Example of a simple localized string
    return $localize \`:ErrorAlert|Generic error message@@common.error.generic:An unexpected error occurred. Please try again.\`;
  }

  constructor() {{'{'}}
    // console.log(this.getWelcomeMessage('Alice'));
    // console.log(this.getErrorMessage());
  }
}

Strings marked with $localize will also be extracted by the ng extract-i18n command.

Angular i18n Workflow and Build Process

Here's a typical workflow for internationalizing your Angular application:

  1. Mark all text for translation: Use the i18n attribute in templates and $localize in your TypeScript code for all user-facing text.
  2. Extract translations to a source file: Use the Angular CLI command: 
    ng extract-i18n --output-path src/locale

    This command will generate a source language file (e.g., src/locale/messages.xlf for XLIFF format). This file contains all the strings you marked for translation.

  3. Create translation files for each locale:
    • Copy the source file (e.g., messages.xlf) for each language you want to support (e.g., messages.fr.xlf for French, messages.es.xlf for Spanish).
    • Translate the text within the <target> elements in these copied files. This is typically done by translators. 
      
<!-- Example snippet from messages.fr.xlf -->
<trans-unit id="home.welcome" datatype="html">
  <source>Welcome to our application!</source>
  <target state="translated">Bienvenue sur notre application !</target> <!-- Translated text -->
  <note priority="1" from="description">User welcome message on the home page</note>
</trans-unit>
  4. Configure angular.json for locales:

    In your angular.json file, you need to define the locales your application supports and specify where the translation files are located. This is done under the projects.[your-project-name].i18n section and in the build configurations.

    
// angular.json (conceptual snippet)
"projects": {{'{'}}
  "[your-project-name]": {{'{'}}
    "i18n": {{'{'}}
      "sourceLocale": "en-US", // Your application's source language
      "locales": {{'{'}}
        "fr": {{'{'}}
          "translation": "src/locale/messages.fr.xlf"
        },
        "es": {{'{'}}
          "translation": "src/locale/messages.es.xlf"
        }
        // Add other locales here
      }
    },
    "architect": {{'{'}}
      "build": {{'{'}}
        "configurations": {{'{'}}
          "production-fr": {{'{'}} // Example for a French production build
            "localize": ["fr"] // Use "fr" configuration from i18n section
            // ... other production settings (output path, etc.)
          },
          "fr": {{'{'}} // Example for a French development build
            "localize": ["fr"]
          }
          // Similar configurations for other locales like "es"
        }
      },
      "serve": {{'{'}}
        "configurations": {{'{'}}
          "fr": {{'{'}}
            "browserTarget": "[your-project-name]:build:fr"
          },
          "es": {{'{'}}
            "browserTarget": "[your-project-name]:build:es"
          }
        }
      }
    }
  }
}

    The localize property in build configurations can also be set to true to build all defined locales, or an array of specific locale IDs.

  5. Build and serve your application for a specific locale: 
    
# Serve the French version for development
ng serve --configuration=fr

# Build the French version for production
ng build --configuration=production-fr

    This will create a version of your application with all marked text replaced by the translations from the corresponding locale file. You would then deploy each localized build to a different URL path (e.g., /fr/, /es/) or subdomain.

Third-Party i18n Libraries

While Angular's built-in i18n is robust for many applications, especially those where a separate build per locale is acceptable, some scenarios require more dynamic internationalization capabilities at runtime (e.g., users switching languages without a page reload or navigating to a different URL).

Popular third-party libraries that offer such features include:

  • @ngx-translate/core: One of the most well-known i18n libraries for Angular. It allows you to load translations at runtime and switch languages dynamically. It uses JSON files for translations by default.
  • @ngneat/transloco: A modern and powerful i18n library for Angular that supports rich features like runtime language switching, multiple translation file formats, and various plugins.

These libraries provide more flexibility for runtime translations but come with their own setup and management considerations. They are often chosen when the compile-time nature of Angular's built-in i18n doesn't fit the project requirements.
