Angular Change Detection

This section explains how Angular detects changes and updates the view, and how you can optimize this process.

What is Change Detection?

Change detection is the mechanism by which Angular synchronizes the state of your application data (the model) with what's displayed in the user interface (the view). When your application data changes, Angular needs to re-render the relevant parts of the view to reflect these changes.

The primary goal of change detection is to ensure that the DOM accurately reflects the current state of the component data.

How Angular Detects Changes (The Default Strategy)

By default, Angular uses a library called Zone.js to determine when to run change detection. Zone.js "monkey-patches" (wraps) most asynchronous browser APIs, such as:

  • setTimeout() and setInterval()
  • DOM events (clicks, mouseovers, keypresses, etc.)
  • XMLHttpRequest (used by HttpClient) and fetch
  • Promises

When any of these asynchronous operations complete, or an event is triggered, Zone.js notifies Angular. Angular then kicks off a change detection cycle for the entire component tree, starting from the root component and checking every component downwards.

The Change Detection Cycle:

During a change detection cycle, Angular performs the following for each component (unless optimized):

  1. Updates input bindings for child components.
  2. Calls lifecycle hooks like ngOnChanges(), ngOnInit() (if it's the first check), and ngDoCheck().
  3. Updates DOM interpolations ({{'{'}}{{'{'}} value }}) in the current component's template.
  4. Recursively performs change detection for child components.
  5. Calls ngAfterViewInit() and ngAfterViewChecked() (and their content equivalents).

This default strategy (often called "dirty checking" or CheckAlways) is convenient because it automatically reacts to most common ways data can change. However, for large applications, running change detection for the entire tree too frequently can lead to performance issues.

Optimizing Change Detection with OnPush

To improve performance, especially in larger applications, Angular provides the ChangeDetectionStrategy.OnPush strategy. When a component uses OnPush, Angular will only run change detection for that component and its subtree if one of the following conditions is met:

  • An @Input() property of the component receives a new reference. Mutating an existing input object/array will NOT trigger change detection for an OnPush component.
  • An event handler (e.g., click, submit) in the component or one of its children is triggered. Angular still runs change detection for the component path after an event.
  • An Observable linked to the template via the async pipe emits a new value. The async pipe internally calls markForCheck().
  • Change detection is explicitly triggered by calling ChangeDetectorRef.detectChanges() or ChangeDetectorRef.markForCheck().

Setting OnPush Strategy:


import {{ '{' }} Component, ChangeDetectionStrategy } from '@angular/core';

@Component({{ '{' }}
  selector: 'app-on-push-example',
  standalone: true,
  template: `<p>OnPush Component. Message: {{'{'}}{{'{'}} data.message }}</p>`,
  changeDetection: ChangeDetectionStrategy.OnPush // Set the strategy here
})
export class OnPushExampleComponent {{ '{' }}
  @Input() data: {{ '{' }} message: string };
}

Using OnPush significantly reduces the number of components Angular has to check during each change detection cycle, as it can skip entire subtrees if their inputs haven't changed by reference.

Immutable Data Structures and OnPush

ChangeDetectionStrategy.OnPush works best with immutable data structures. Immutability means that once an object or array is created, it cannot be changed. If you need to modify it, you create a new object or array with the changes, rather than mutating the original.

Why Immutability Matters for OnPush:

  • Reference Checks: OnPush components rely on reference changes for their @Input() properties. If you pass an object and mutate one of its properties, the reference to the object itself doesn't change, so Angular won't detect a change on an OnPush component.
  • Predictability: Immutable data makes it easier to track changes and understand how state evolves over time.
  • Performance: Comparing object references is much faster than deep-comparing object properties.

Example: Updating an Input for an OnPush Component


// Parent Component
// this.user = {{ '{' }} name: 'Alice', age: 30 }; // Initial user object

// Incorrect way (mutating):
// this.user.age = 31; // OnPush child component will NOT update

// Correct way (creating a new object - new reference):
// this.user = {{ '{' }} ...this.user, age: 31 }; // OnPush child component WILL update

// For arrays:
// this.items = ['a', 'b'];

// Incorrect way (mutating):
// this.items.push('c'); // OnPush child component will NOT update if it receives 'items' as input

// Correct way (creating a new array):
// this.items = [...this.items, 'c']; // OnPush child component WILL update

Libraries like Immer or Immutable.js can help manage immutable state, but even using spread syntax for objects ({{'{'}}...obj}) and array methods that return new arrays (.map(), .filter(), spread syntax [...arr]) is a good practice.

Manually Triggering Change Detection

Sometimes, you might need to tell Angular explicitly to run change detection, especially when working with OnPush components or when changes happen outside of Angular's zone (e.g., from a third-party library that Angular doesn't patch).

You can inject ChangeDetectorRef into your component:


import {{ '{' }} Component, ChangeDetectionStrategy, ChangeDetectorRef, Input } from '@angular/core';

@Component({{ '{' }}
  selector: 'app-manual-cd-example',
  standalone: true,
  template: `<p>Value: {{'{'}}{{'{'}} value }}</p>`,
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class ManualCdExampleComponent {{ '{' }}
  @Input() value: number = 0;

  constructor(private cdr: ChangeDetectorRef) {{ '{' }} }

  updateValueProgrammatically(newValue: number) {{ '{' }}
    // Imagine this method is called by a non-Angular async operation
    this.value = newValue;
    // If this component were OnPush and the input reference didn't change,
    // or if 'value' was an internal property changed outside Angular's zone,
    // the view might not update without manual intervention.
  }

  // Option 1: Run change detection for this component and its children.
  forceDetectChanges() {{ '{' }}
    this.updateValueProgrammatically(Math.random());
    this.cdr.detectChanges(); // Runs CD for this component and its subtree. Use sparingly.
  }

  // Option 2: Mark this component and its ancestors as dirty, to be checked in the next CD cycle.
  forceMarkForCheck() {{ '{' }}
    this.updateValueProgrammatically(Math.random());
    this.cdr.markForCheck(); // Marks the path from this component to root as needing check.
                           // Relies on the next CD cycle (triggered by Zone.js or manually).
  }
}

detectChanges() vs markForCheck():

  • detectChanges(): Runs change detection for the current component and its children immediately. It does not check ancestors. Use this if you are sure the change is isolated to this component's subtree. Can be heavy if overused.
  • markForCheck(): Marks the component and all its ancestors as "dirty" (needing a check). It does not schedule or run change detection itself. Instead, it ensures that when the next change detection cycle runs (either triggered by Zone.js or an explicit call to ApplicationRef.tick() or detectChanges() on a parent/root), this component will be checked, even if it's OnPush and its inputs haven't changed reference. This is generally safer and more efficient than detectChanges() for OnPush components.

Understanding these concepts allows you to build highly performant Angular applications by taking control of how and when change detection runs.
